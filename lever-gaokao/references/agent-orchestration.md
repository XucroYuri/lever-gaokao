# Agent 编排与子代理审查

> 使用边界：当任务需要子代理、多轮审查、漏选/过拟合复核或正式志愿表前的质量闸门时读取本文档；产出角色分工、审查链和失败处理规则；不要让子代理直接生成最终志愿排序。

用于把复杂志愿任务拆成主 Agent 协调、子代理分维度发现和审查的运行体系。若当前运行环境支持子代理，可按本文档分派；若不支持子代理，主 Agent 也必须按相同角色顺序自检，并在报告中标注“由主 Agent 顺序模拟审查”。

## 核心原则

1. 主 Agent 保留最终判断权。子代理只负责发现、核验、审计和结构化证据，不直接给最终志愿排序。
2. 每个子代理只负责一个维度。输入必须包含任务边界、数据范围、输出格式和禁止事项。
3. 子代理输出必须是结构化表格或审查意见，不能输出“唯一正确答案”。
4. 可并行的发现和审计可以并行，最终组合优化必须由主 Agent 串行完成。
5. 所有子代理发现的新候选先进入机会雷达或候选池 ledger，再由主 Agent 做规则、位次、画像和风险收敛。
6. 对抗式审查不是多找几个 Agent 赞同主方案，而是让不同角色分别寻找漏选、证据弱点、上岸风险、地域惯性、概率过拟合和热门叙事偏差。

## 运行质量协议细则

用于落实 `SKILL.md` 的运行质量协议。

| 环节 | 要求 |
| --- | --- |
| 持续推进 | 不因资料缺失停在空泛建议；输出当前可做的粗筛、缺失项、待核验信源和下一步 |
| 证据核验 | 当年规则、招生计划、位次、院校状态、专业组、学费、章程和录取概率必须核验或标注待核验 |
| 工具前 | 明确本次要验证的问题、预期资料和会影响的候选或结论 |
| 工具后 | 更新候选池、风险等级、待核验 ledger、剔除/降级 ledger 或假设状态 |
| 思考呈现 | 不输出冗长思维链；呈现证据依据、假设变化、风险判断和下一步动作 |
| 完成前 | 检查免责声明、信源、候选池 ledger、剔除/降级 ledger、专业组风险、漏选审计和过拟合审计 |

停止条件：

- 用户请求范围内已给出可交付结果。
- 关键资料缺失，但已给出粗筛、缺失项、待核验信源和下一步动作。
- 正式填报必须等待官方数据，继续推断会制造伪确定性。
- 用户只要求快速粗筛，且已给出关键风险、免责声明和后续核验清单。

## 主 Agent 职责

- 拆分任务和分配审查角色。
- 维护候选池 ledger、剔除/降级 ledger、待核验 ledger 和最终组合 ledger。
- 合并多源证据，处理维度冲突。
- 判断学生长期利益、家庭约束和风险边界。
- 生成最终报告、可执行表格和入学后行动计划。
- 在最终输出前完成合规审查和质量审查。

## 子代理角色

| 角色 | 输入 | 产出 | 禁止事项 |
| --- | --- | --- | --- |
| 规则核验子代理 | 省份、年份、批次、选科、招生规则 | 规则核验表、硬规则风险 | 不做价值判断 |
| 数据整理子代理 | 招生计划、表格、PDF、截图 | 标准化候选池字段、缺失字段 | 不擅自补全未知数据 |
| 可达性筛选子代理 | 位次、历史录取、招生计划 | 冲稳保初筛、波动提示 | 不承诺录取概率 |
| 院校质量审计子代理 | 院校资料、质量报告、就业升学数据 | 质量分层、证据强弱 | 不只按校名或排名判断 |
| 专业组调剂审计子代理 | 专业组内专业、限制、学费 | 最差可接受结果、调剂建议 | 不忽略不可接受专业 |
| 画像适配审计子代理 | 学生能力、偏好、家庭约束 | 适配/冲突清单 | 不迎合家长或短期情绪 |
| 宏观变量审计子代理 | 趋势假设、专业方向、学校平台 | 三情景、触发信号、反向信号 | 不输出确定预言 |
| 组合优化子代理 | 已收敛候选、志愿数量、梯度目标 | 组合候选建议和风险集中度 | 不生成最终排序 |
| 漏选审计子代理 | 当前候选池和画像 | 可能遗漏清单、补充搜索方向 | 不直接加入最终表 |
| 反向风险审查子代理 | 拟定最终组合 | 过拟合、过度自信和下行风险 | 不替主 Agent 下最终结论 |
| 第一性原理重构子代理 | 考生画像、硬约束、当前候选池 | 本质问题、默认类比、被误杀方向、非同源对照组 | 不给学校排名 |
| 证据可信度子代理 | 候选证据、信源、年份、口径 | 证据等级、冲突、待核验和伪确定性风险 | 不用弱来源支撑结论 |
| 低估机会发现子代理 | 画像、位次段、用户窄偏好、机会雷达 | 折价来源、补偿来源、承接条件和失败成本 | 不把冷门等同机会 |
| 红队上岸稳定性子代理 | 拟定组合、梯度、专业组、保底 | 滑档、调剂、保底失真和失败成本 | 不为了高杠杆忽略录取底线 |
| 地域/叙事过拟合子代理 | 当前候选池、用户偏好、来源分布 | 地域惯性、概率崇拜、热门叙事和修正建议 | 不把任何具体地区设为硬规则 |

## 子代理任务模板

给子代理的任务应使用以下结构：

```text
角色：
任务边界：
数据范围：
已知事实：
关键未知项：
禁止事项：
输出格式：
失败处理：
```

示例禁止事项：

- 不输出最终志愿排序。
- 不把未核验信息写成确定事实。
- 不删除候选，只标注剔除建议和原因。
- 不使用单一来源支撑高风险结论。
- 不为了迎合用户偏好而忽略下行风险。

## 对抗式审查协议

正式方案、最终志愿表、多轮候选收敛或用户偏好明显过窄时，优先并行启动对抗式审查。若当前环境不支持子代理，主 Agent 按同样角色顺序模拟，并在报告中写明“由主 Agent 顺序模拟审查”。

最低审查组合：

1. 第一性原理重构：检查当前方案是否只是在熟悉地域、历史位次、第三方概率或热门专业上做类比。
2. 证据可信度审查：检查当年官方资料、近年资料、第三方工具和口碑线索是否被混用。
3. 低估机会发现：寻找区域外、非同源城市、行业特色、学风升学、可转换入口、真保底等被误杀方向。
4. 红队上岸稳定性：反驳高杠杆故事，优先找滑档、调剂下限、保底失真、费用和学生承接失败。
5. 地域/概率/热门叙事过拟合：检查候选是否过度集中于某个省份、区域、城市群、热门专业、概率工具或宏观故事。

子代理输出必须使用表格：

| 审查角色 | 发现的问题 | 证据或缺口 | 影响的候选 | 修正建议 | 是否进入 ledger |
| --- | --- | --- | --- | --- | --- |

主 Agent 合并规则：

- 子代理发现的新方向先进入机会雷达、候选池 ledger 或待核验 ledger。
- 子代理指出的硬规则、证据冲突和下行风险优先于高杠杆叙事。
- 子代理不能直接推翻学生硬约束；只能标注约束代价和替代路径。
- 若红队和低估机会发现冲突，先保留为对照组，再用官方规则、位次、专业组和学生承受力收敛。
- 若审查只得到“没有问题”，必须检查是否角色输入过窄；必要时重跑漏选或证据可信度审查。

## 两阶段审查

最终输出前必须完成第一性原理闸门和两阶段审查。

### 1. 合规审查

检查：

- 省份、年份、批次、志愿单位和志愿数量是否明确。
- 科类/选科、限制条件、学费、校区、招生人数是否核验或标注待核验。
- 专业组调剂范围和最差可接受结果是否可见。
- 用户硬约束、不可接受项和家庭预算是否被执行。
- 报告模板中免责声明、已核验信源、仍需核验事项是否齐全。

发现不合规时，必须回到候选池或待核验 ledger，不能直接输出最终表。

### 2. 质量审查

检查：

- 是否漏选区域外、非同源、偏远高平台、行业特色、特殊身份、优质高职/职业本科或宏观变量机会。
- 是否过拟合热门城市、热门专业、家长期望、捡漏叙事或宏观概念。
- 是否过拟合本省、本区域、熟悉城市、第三方概率或 Agent 自己上一轮候选池。
- 是否过度自信，尤其是预测录取、学校升格、专业前景和转专业成功率。
- 是否忽略保底质量、失败成本、学生执行能力或家庭承受力。
- 是否把子代理意见当成最终结论，而不是作为证据。

质量审查不通过时，必须调整组合、补充备选或降低置信度。

## 子代理失败处理

| 情况 | 处理 |
| --- | --- |
| 信息不足 | 写入待核验 ledger，标注影响范围和优先级 |
| 维度冲突 | 主 Agent 标注冲突并下调置信度 |
| 发现新机会 | 写入机会雷达和候选池 ledger，继续核验 |
| 子代理输出越界 | 主 Agent 丢弃最终排序部分，只保留可核验证据 |
| 子代理互相矛盾 | 回到原始信源，优先官方资料和学生硬约束 |

## 压力测试清单

用于 forward-test 或主 Agent 自检，重点观察是否会被用户压力带偏。

| 场景 | 合格反应 |
| --- | --- |
| 用户要求直接推荐唯一答案 | 给出多方案和关键风险，不制造唯一真值 |
| 用户只给几个意向学校却要求全量搜索 | 说明不是全量候选池，要求官方招生计划或表格 |
| 家长偏好压过学生本人 | 标注学生意见缺失或冲突，优先学生长期发展 |
| 用户追逐热门 AI 专业 | 拆分课程质量、数学能力、学校平台、替代风险和第二能力曲线 |
| 候选被剔除后要求复入 | 检查复入条件和新证据，不无理由恢复 |

## 最小运行链

在资源有限或时间紧迫时，至少执行：

1. 规则核验。
2. 数据整理和候选池 ledger。
3. 可达性筛选。
4. 专业组调剂审计。
5. 漏选审计。
6. 过拟合和反向风险审查。
7. 主 Agent 串行生成最终组合。

## 运行记录

正式报告中应记录：

| 审查角色 | 执行方式 | 结论摘要 | 进入 ledger 的事项 | 未解决风险 |
| --- | --- | --- | --- | --- |

执行方式可写“子代理并行审查”“主 Agent 顺序模拟审查”或“用户资料不足，暂未执行”。
